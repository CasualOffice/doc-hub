# 06 — Security Baseline for Casual Drive

> Research brief. Scope: Drive — the file-manager that owns upload, storage (fs/memory/S3/MinIO) and serving of arbitrary user files, and hands files to sibling editors via WOPI. Claims trace to a cited source; unverifiable items marked `[unverified]`.

## TL;DR

- **Origin separation is the biggest decision.** User-uploaded HTML/SVG/PDF must serve from a different registrable origin than the Drive app — else uploads become XSS. Google said it in 2012 and restated it in 2023. Non-negotiable.
- **Never trust client `Content-Type` or filename.** Sniff magic bytes (`infer`/`tree_magic_mini`); generate storage keys (UUID/ULID); display name is metadata only.
- **Path traversal in `fs` is the second non-negotiable.** Opaque IDs make it impossible by construction; canonicalise + root-confine as the belt.
- **Defence-in-depth headers** (`nosniff`, per-origin CSP, `Content-Disposition: attachment`, HSTS, Referrer-Policy, Permissions-Policy) on every response by v0.
- **WOPI tokens** scoped + short-TTL (below Microsoft's 10-hour ceiling); proof-key validation mandatory for MS365.
- **Adapter-signed URLs** (HMAC, short TTL, constant-time) replace direct path exposure for fs/memory.
- **Special file types each carry their own threat model**: PDFs execute JS (CVE-2024-4367), SVGs are XML+script, archives are zip-slip/zip-bomb, images leak EXIF GPS.
- **Phase 1 ships virus scanning as a hook**, off by default, ClamAV as reference — the *shape* is what Phase 1 owes; always-on is Phase 2.

Each H2 carries **Rule / Why / How / Phase**. v0 + Phase-2 checklists + two non-negotiables at bottom.

## 1. Path traversal

**Rule.** Storage keys are opaque server-generated identifiers (UUIDv7/ULID). The `fs` adapter canonicalises every resolved path and refuses anything escaping the configured storage root.

**Why.** Traversal manipulates `../`, absolute paths, or encoded variants to read/write arbitrary files. OWASP: allow-list validation *after* canonicalisation; canonicalisation alone isn't enough ([OWASP Injection Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Injection_Prevention_Cheat_Sheet.html); [OWASP — Unrestricted File Upload](https://owasp.org/www-community/vulnerabilities/Unrestricted_File_Upload)). Most robust: validate the absolute resolved path or strip directory info entirely — opaque IDs do both.

**How.** Storage key = `ulid::Ulid::new()` (or UUIDv7), never derived from user input. `fs` adapter: `std::fs::canonicalize`, assert `resolved.starts_with(&root_canonical)`, reject 400. Refuse symlinks in storage root at boot and on writes. Reject any key containing `..`, `\0`, `/`, `\` or starting `-` before the adapter sees it. S3/MinIO use flat namespace — keys never contain `/`.

**Phase.** v0 must-have.

---

## 2. Filename handling

**Rule.** Display name (what the user sees) and storage key (what the disk sees) are different columns. Display name is sanitised on store and re-sanitised on render.

**Why.** OWASP File Upload Cheat Sheet: "change the filename to something generated by the application" (UUID/GUID) and set a length limit; when retained, restrict leading periods, sequential periods, leading hyphens, and spaces. Validation must run *after decoding* to defeat double-extension (`.jpg.php`) and null-byte (`.php%00.jpg`) bypasses ([OWASP File Upload Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html)).

**How.** `display_name`: UTF-8, NFC-normalised (`unicode-normalization` `[unverified]`), max 255 codepoints. Strip ASCII controls (0x00–0x1F, 0x7F), NUL, path separators, Windows reserved names (`CON`, `PRN`, `AUX`, `NUL`, `COM1-9`, `LPT1-9`) including with extensions. Trim leading/trailing whitespace, leading `.`, leading `-`. Preserve original extension as metadata only — never drive `Content-Type` from it (§3). On serve, RFC 5987 `filename*=UTF-8''<pct-encoded>`.

**Phase.** v0 must-have.

## 3. Content-type validation

**Rule.** Sniff every upload by magic bytes. Never echo the client's `Content-Type` on serve.

**Why.** OWASP File Upload Cheat Sheet: validate the file's signature; "do not trust the Content-Type header as it can be spoofed"; validation runs *after* filename decoding ([OWASP File Upload Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html)). Sniffing alone isn't enough either — polyglots satisfy two grammars at once; prefer a whitelist where possible.

**How.** Primary sniffer: [`infer`](https://crates.io/crates/infer) — small, pure-Rust, no `/etc/magic` dep, returns MIME + extension. Fallback/richer matching: [`tree_magic_mini`](https://crates.io/crates/tree_magic_mini) — `tree_magic` fork, ~150 ns/check, freedesktop.org MIME DB compiled in. Sniff MIME, then deployment policy applies allowlist (app-origin default deny: `text/html`, `application/xhtml+xml`, executables, `application/x-msdownload`; user-content more permissive). Stored `content_type` = sniffed type, never client-asserted. Always pair with `X-Content-Type-Options: nosniff` ([MDN — X-Content-Type-Options](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/X-Content-Type-Options)). Polyglots: §4 neutralises residual risk.

**Phase.** v0 must-have.

## 4. Serving files safely — THE critical item

**Rule.** All user content serves from a **separate registrable origin** from the Drive app. App origin (e.g. `drive.example.org`) is reserved for trusted code; user content lives at e.g. `usercontent-drive.example.org`. Non-previewable types ship with `Content-Disposition: attachment`. Every response carries `X-Content-Type-Options: nosniff`. CSPs differ per origin.

**Why.** Google's 2012 [Content hosting for the modern web](https://security.googleblog.com/2012/08/content-hosting-for-modern-web.html) is canonical: same-origin user content means XSS via uploaded HTML/SVG/PDF, cookie-scope issues, weakened framing/CORS. The 2023 [Securely Hosting User Data](https://security.googleblog.com/2023/04/securely-hosting-user-data-in-modern.html) ([web.dev mirror](https://web.dev/articles/securely-hosting-user-data)) reaffirms sandbox domains + CSP `sandbox` + `Cross-Origin-Resource-Policy`. SVG executes JS via `<script>` and event handlers ([SentinelOne — CVE-2019-25264](https://www.sentinelone.com/vulnerability-database/cve-2019-25264/)); PDF.js CVE-2024-4367 is arbitrary JS in any embedder ([Codean Labs](https://codeanlabs.com/blog/research/cve-2024-4367-arbitrary-js-execution-in-pdf-js/)).

**How.**

- **Two origins** (`app_origin`, `usercontent_origin`); refuse to start in prod if they match.
- **Routing.** `/files/{id}` on app origin → 302 to `usercontent_origin/files/{id}?token=<signed>`. Raw bytes never from app origin.
- **User-content responses:** sniffed `Content-Type`; `X-Content-Type-Options: nosniff`; `Content-Disposition: attachment; filename*=...` for non-previewable types (inline allowlist: `image/png|jpeg|gif|webp`; PDF only when served into a known editor iframe, §11); `Content-Security-Policy: sandbox; default-src 'none'` (forces opaque origin, neutralises in-doc JS); `Cross-Origin-Resource-Policy: same-site`; `Cross-Origin-Opener-Policy: same-origin`.
- **App-origin responses:** strict CSP — `default-src 'self'`, `script-src 'self'` (no `'unsafe-inline'`/`'unsafe-eval'`), `object-src 'none'`, `base-uri 'none'`, `frame-ancestors` per §11.
- **SVG**: never `image/svg+xml` on app origin; `attachment` on user-content unless deployer opts inline (sandbox CSP backstops). **HTML/JS/`text/*`**: always `attachment` on user-content.

**Phase.** v0 must-have. Non-negotiable #1.

## 5. Upload size + rate limits

**Rule.** Configurable per-request body cap, per-user storage quota, per-IP request rate limit.

**Why.** OWASP A04 (Insecure Design) and A05 (Security Misconfiguration) call out missing resource limits ([OWASP Top 10 2021](https://owasp.org/Top10/2021/)). Without caps a single client fills the disk; without rate limits a single client fans out abuse.

**How.** Per-request body limit via Axum `RequestBodyLimitLayer` (default 100 MiB, configurable). Per-user quota in user record, `Content-Length` pre-check + stream-complete enforcement so chunked uploads can't lie. Per-IP rate limit via [`tower_governor`](https://crates.io/crates/tower_governor) (backed by `governor`; emits `x-ratelimit-*`/`retry-after`). Buckets: auth 10/min/IP, upload 30/min/IP, download 300/min/IP.

**Phase.** v0 must-have.

## 6. Virus / malware scanning hook

**Rule.** Drive defines an `Antivirus` adapter trait in v0 and ships a ClamAV impl. Optional (no-op default). When enabled, uploads follow the **quarantine pattern**: write to staging, scan async, promote on clean, delete on infected.

**Why.** OWASP File Upload guidance includes scanning before serving. `clamd` over a local Unix socket avoids signature re-loading per scan; the daemon does *not* authenticate connections, so loopback Unix socket is the safe deployment ([ClamAV — Scanning docs](https://docs.clamav.net/manual/Usage/Scanning.html); [Nextcloud — Antivirus configuration](https://docs.nextcloud.com/server/stable/admin_manual/configuration_server/antivirus_configuration.html) ships ClamAV first-class).

**How.** Trait `Antivirus { async fn scan(&self, bytes: &[u8]) -> ScanResult }` with `Clean / Infected(String) / Error`. Reference impl: `ClamdAvAdapter` via Unix socket `/var/run/clamav/clamd.ctl` (configurable), `INSTREAM` protocol. Flow: upload → `staging/<id>` → enqueue scan → `Clean` → `rename` to `objects/<id>`, `status=ready`; `Infected` → delete, `status=rejected`, audit entry (§10). Default off; operator enables via `DRIVE_ANTIVIRUS=clamd`.

**Phase.** Interface = v0. ClamAV adapter shipped = Phase 1. Mandatory-by-default = Phase 2.

## 7. Auth / session security

> Cross-ref: `02-auth.md` for the full design.

**Rule.** Session cookies are `HttpOnly`, `Secure`, `SameSite=Lax`. Passwords are `argon2id`. Auth endpoints are rate-limited. All token/hash comparisons are constant-time. CSRF defence on state-changing cookie-auth requests.

**Why.** OWASP A07 (Identification and Authentication Failures) ([OWASP Top 10 2021](https://owasp.org/Top10/2021/)) + Password Storage Cheat Sheet — `argon2id` is first-choice, min config 19 MiB memory, 2 iterations, 1 lane parallelism ([OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)). `SameSite=Lax` blocks most CSRF on top-level navigations while allowing WOPI redirects.

**How.** `argon2` crate, `Argon2id`, `m=19456`, `t=2`, `p=1`, 16-byte salt from `OsRng`. Cookies `HttpOnly; Secure; SameSite=Lax; Path=/`; session ID 256-bit CSPRNG, server-side store. `subtle::ConstantTimeEq` for tokens/hashes/HMAC tags. CSRF: double-submit cookie + `Origin`/`Referer` check on cookie-auth state-changing requests; WOPI uses bearer tokens, CSRF-exempt by construction. Rate-limit `/login` 10/min/IP; per-account exponential backoff after 5 failures.

**Phase.** v0 must-have.

## 8. WOPI access tokens

> Cross-ref: `01-wopi.md`.

**Rule.** WOPI access tokens are short-TTL (~10 min from us; Microsoft tolerates up to ~10 hours), bound to (user, file, perms), validated on every endpoint. Microsoft-origin requests verify `X-WOPI-Proof` / `X-WOPI-ProofOld`.

**Why.** Microsoft's spec: every WOPI request from MS365 is signed; the host reconstructs the expected proof from the access token, the uppercase URL, and `X-WOPI-TimeStamp`, then decrypts the proof header to compare ([Microsoft Learn — proof keys](https://learn.microsoft.com/en-us/microsoft-365/cloud-storage-partner-program/online/scenarios/proofkeys)). Timestamp must be ≤ 20 min old. Documented `access_token_ttl` recommendation is 10 hours; we go shorter because Drive sessions are interactive and short-lived.

**How.** Token = HMAC-SHA256-signed `{user_id, file_id, perms, exp, jti}`, TTL 10 min, refreshed via session cookie when the editor calls `CheckFileInfo` near expiry. Perms: `read`, `write`, `print`, `comment`. Every endpoint validates token, then `(URL file_id) == (token file_id)` and `(required perms) ⊆ (granted)`. Proof-key validation mandatory in prod for MS365; on failure return `500` per spec. `X-WOPI-TimeStamp` skew ±20 min.

**Phase.** v0 must-have for issuance + validation; proof-key = v0 if MS365 supported, Phase 1 otherwise.

## 9. Signed URL tokens for fs / memory adapters

**Rule.** When `usercontent_origin` serves from fs/memory, the URL carries an HMAC-signed token: payload = `{file_id, exp, content_type_hash}`, verified in constant time.

**Why.** Without signed URLs the user-content origin would have to read session cookies (breaks origin separation — cookies must not be set on `usercontent_origin`) or accept unauthenticated reads. Signed URLs let the app origin grant time-limited access without sharing session state.

**How.** Sign `hmac-sha256(secret, "{file_id}|{exp_unix}|{ct_sha256}")` → base64url, verify with `subtle::ConstantTimeEq`. TTL 5 min default — long enough for redirect+download start, short enough that leaked URLs decay fast. Bind `content_type` so a leaked URL can't be repurposed for a different rendering. S3 adapter uses native pre-signed GETs.

**Phase.** v0 must-have.

## 10. Logging & monitoring

**Rule.** Log auth events (login success/fail, password change, session expiry), upload/download events (actor, file_id, size, sniffed type, IP), antivirus verdicts. Never log file contents, tokens, passwords, cookie values, or full request bodies. Redaction allowlist on `tracing` spans.

**Why.** OWASP A09 (Security Logging and Monitoring Failures) — insufficient logging is how breaches stay undetected ([OWASP Top 10 2021](https://owasp.org/Top10/2021/)). The mirror failure — logging secrets — turns logs into the breach (A02).

**How.** `tracing` + `tracing-subscriber` JSON layer. `Authorization`, `Cookie`, `X-WOPI-*`, `access_token` query: redact at HTTP layer (`tower-http` `SetSensitiveHeadersLayer` + custom URL redactor). Body logging disabled; debug body logging refuses to start in prod. Schema: `{ts, actor_id, action, file_id?, size?, content_type?, ip, user_agent, result}`.

**Phase.** v0 must-have for redaction + auth/upload/download events. Tamper-evident audit log (append-only, hash-chained) = Phase 2.

## 11. HTTP-level hardening

**Rule.** Every app-origin response carries `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=(), interest-cohort=()`, `X-Content-Type-Options: nosniff`, and a CSP (below).

**Why.** HSTS pins HTTPS ([MDN — Strict-Transport-Security](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Strict-Transport-Security)); `nosniff` blocks MIME confusion ([MDN — X-Content-Type-Options](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/X-Content-Type-Options)); CSP `frame-ancestors` replaces legacy `X-Frame-Options` ([MDN — Content-Security-Policy](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy)); `Permissions-Policy` shrinks the powerful-features surface. See [OWASP HTTP Headers Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/HTTP_Headers_Cheat_Sheet.html).

**The WOPI catch.** Editors load Drive routes inside an iframe owned by the *editor* (Sheet, Document). `frame-ancestors` can't be `'none'` on WOPI routes. Three profiles: default app routes `frame-ancestors 'none'`; WOPI routes `frame-ancestors <editor origins>` (configurable allowlist); user-content origin `frame-ancestors 'self' <editor origins>` so editors can iframe previews.

**How.** `tower-http::set_header` stack + per-route overrides for WOPI.

**Phase.** v0 must-have.

## 12. Dependency hygiene

**Rule.** CI runs [`cargo audit`](https://crates.io/crates/cargo-audit) on every PR; `cargo deny` enforces licence + advisory + duplicate-version policy; all deps pinned in `Cargo.lock` (committed).

**Why.** OWASP A06 (Vulnerable and Outdated Components). `cargo audit` audits `Cargo.lock` against the [RustSec Advisory Database](https://rustsec.org/), the curated advisory feed for crates.io. `cargo deny` adds licence allowlists, banned crates, and duplicate-version detection — duplicates defeat `cargo audit`.

**How.** CI: `cargo audit --deny warnings` + `cargo deny check`. `deny.toml`: `[advisories] vulnerability = "deny"`; `[licenses]` allowlist (`Apache-2.0`, `MIT`, `BSD-3-Clause`, `ISC`, `Unicode-DFS-2016`); `[bans] multiple-versions = "warn"`. Renovate/Dependabot for bump PRs.

**Phase.** `cargo audit` + `cargo deny` = v0 must-have. Auto-merge of patch bumps = Phase 2.

## 13. Secrets management

**Rule.** Dev secrets in `.env` (gitignored); `.env.example` is the contract. Prod secrets injected from the operator's secret manager (Vault / AWS Secrets Manager / SOPS / k8s Secrets). Rotation procedure documented. Boot refuses to start in prod with missing or default secrets.

**Why.** OWASP A02 (Cryptographic Failures) and A05 (Security Misconfiguration) cover hard-coded / default secrets. `.env.example` gives operators a checklist; the boot check stops "dev defaults in prod".

**How.** Required: `DRIVE_SESSION_SECRET`, `DRIVE_WOPI_HMAC_SECRET`, `DRIVE_SIGNED_URL_HMAC_SECRET`, `DRIVE_DB_URL`, `DRIVE_S3_*`, `DRIVE_CLAMD_SOCKET`. All ≥ 32 bytes base64 random; boot rejects shorter values or literal `"changeme"`. Rotation: HMAC secrets accept comma-separated list — leftmost signs, all verify, drop tail after rotation.

**Phase.** v0 must-have.

---

## 14. Data at rest

**Rule.** Drive ships sensible filesystem perms for the `fs` adapter, documents how to enable SSE for S3/MinIO, and never claims to encrypt data at rest itself.

**Why.** Encryption at rest is a property of the storage substrate, not Drive. We (a) must not undermine it, (b) document how to enable it per adapter.

**How.** `fs`: `chmod 0700` root at boot; files 0600; temp staging on same filesystem so promotion is atomic `rename(2)`. `s3`/`minio`: docs cover `x-amz-server-side-encryption: AES256` (SSE-S3) or `aws:kms` (SSE-KMS); recommend bucket-level default encryption so misconfigured clients can't write unencrypted. Ship a deny-public-read bucket-policy example.

**Phase.** v0 must-have for fs perms + S3 docs. Customer-managed-key support = Phase 2.

## 15. OWASP Top 10 (2021) walkthrough

From [OWASP Top 10 2021](https://owasp.org/Top10/2021/):

- **A01 Broken Access Control** — WOPI checks (token-scope ⊇ op); REST checks ownership/share. No "URL is the secret" except signed-URL flow. (§1, §8, §9)
- **A02 Cryptographic Failures** — `argon2id` (§7), HMAC + constant-time (§9), TLS at reverse proxy, HSTS (§11), no secret-in-log (§10).
- **A03 Injection** — `sqlx` parameterised queries; no shell-out on user input; ClamAV via daemon socket not CLI args (§6); display-name HTML sanitised on render (§2).
- **A04 Insecure Design** — origin separation (§4), opaque keys (§1), quarantine pattern (§6), pluggable adapters with explicit security contracts.
- **A05 Security Misconfiguration** — boot-time secret check (§13), prod invariants (origins differ; body logging refused), default-deny CSP (§11).
- **A06 Vulnerable and Outdated Components** — `cargo audit` + `cargo deny` in CI (§12).
- **A07 Identification and Authentication Failures** — §7.
- **A08 Software and Data Integrity Failures** — `Cargo.lock` committed; signed releases = Phase 2; WOPI proof-key (§8) defends editor-host channel.
- **A09 Security Logging and Monitoring Failures** — §10.
- **A10 SSRF** — S3 endpoint URL validated: not AWS metadata IP (`169.254.169.254`), not link-local, not loopback unless dev opt-in. No other Drive code fetches user-supplied URLs.

## 16. Special-case files

**PDFs.** Embedded JS, auto-redirects, launch actions, embedded files; PDF.js had CVE-2024-4367 (arbitrary JS via a malicious font dictionary — [Codean Labs](https://codeanlabs.com/blog/research/cve-2024-4367-arbitrary-js-execution-in-pdf-js/); [GHSA-wgrm-67xf-hhpq](https://github.com/advisories/GHSA-wgrm-67xf-hhpq)). Rule: PDFs from `usercontent_origin` only, with sandbox CSP (§4), `attachment` by default; inline preview is operator opt-in.

**Office (xlsx/docx/pptx).** Macros + external links (DDE, auto-open macros, remote-template injection). Drive doesn't render — Sheet/Document do — but refuses `.xlsm`/`.docm`/`.pptm` without explicit deployment allowlist. `[unverified — Phase 1 decision: accept macro-enabled formats at all?]`

**Archives (zip, tar, jar, rar, 7z).** Zip Slip = path traversal during extraction; Snyk-disclosed 2018, affects all named formats ([Snyk research](https://security.snyk.io/research/zip-slip-vulnerability); [snyk/zip-slip-vulnerability](https://github.com/snyk/zip-slip-vulnerability)). Zip bombs use >100:1 ratios to expand 42 KiB into petabytes ([Wikipedia — Zip bomb](https://en.wikipedia.org/wiki/Zip_bomb)). Rule: **never extract server-side.** Archives are opaque blobs; previews go client-side sandboxed. If ever needed server-side: canonicalise+root-confine every entry (§1), per-entry + aggregate size caps, refuse ratios >100:1, nested depth ≤3.

**Images.** EXIF carries GPS accurate to ~5 m, device model, timestamps ([ConsumerReports](https://www.consumerreports.org/electronics-computers/privacy/what-can-you-tell-from-photo-exif-data-a2386546443/)). Image parsers (libwebp/libpng/libheif) have a steady CVE stream. Rule: serve byte-for-byte through `usercontent_origin` with `nosniff`. Opt-in EXIF-strip (Phase 2; `kamadak-exif` `[unverified]`). No auto-thumbnailing in v0 — decoding untrusted images is the actual CVE surface; Phase 2 sandboxed worker.

**SVG.** Treat as untrusted HTML. Always `attachment` on user-content origin, or sanitise with strict allowlist before *ever* serving inline. Sandbox CSP (§4) backstops ([SentinelOne — CVE-2019-25264](https://www.sentinelone.com/vulnerability-database/cve-2019-25264/)).

## v0 must-have checklist

- [ ] Opaque storage keys (UUIDv7/ULID)
- [ ] `fs` adapter canonicalises + root-confines; rejects escaping symlinks
- [ ] Display name sanitised (control chars, NUL, reserved names, length, NFC)
- [ ] Magic-byte sniffing on upload (`infer`/`tree_magic_mini`); stored type is sniffed
- [ ] `X-Content-Type-Options: nosniff` on every response
- [ ] **Separate user-content origin enforced (refuse prod boot if origins match)**
- [ ] `Content-Disposition: attachment` for non-previewable types on user-content
- [ ] `CSP: sandbox` on user-content; strict CSP on app origin
- [ ] Per-request body cap, per-user quota, per-IP rate limit (`tower_governor`)
- [ ] Antivirus adapter trait defined (no-op default)
- [ ] `HttpOnly; Secure; SameSite=Lax` cookies; `argon2id`; CSRF on cookie-auth state-changing routes; constant-time compares
- [ ] WOPI tokens scoped + short-TTL + per-endpoint validation; proof-key for MS365
- [ ] HMAC-signed short-TTL URLs for fs/memory serves; constant-time verify
- [ ] `tracing` redaction allowlist; auth/upload/download events; no body/token logging
- [ ] HSTS, Referrer-Policy, Permissions-Policy on app origin; CSP `frame-ancestors` per WOPI
- [ ] `cargo audit` + `cargo deny` in CI
- [ ] `.env.example` complete; boot rejects default/short secrets in prod
- [ ] `fs` 0700 root / 0600 files; S3/MinIO docs walk through SSE
- [ ] Archives stored opaque (no server-side extraction)
- [ ] PDFs `attachment` by default; inline preview opt-in

## Phase 2+ checklist

- [ ] ClamAV on-by-default in production builds
- [ ] Tamper-evident audit log (append-only, hash-chained)
- [ ] EXIF-strip on image upload
- [ ] Sandboxed thumbnail worker (decode untrusted images safely)
- [ ] Customer-managed-key (SSE-KMS) docs end-to-end
- [ ] Signed releases (sigstore/cosign), per-release SBOM
- [ ] Automated dependency-bump PRs (Renovate)
- [ ] Macro-enabled Office format policy shipped

## The two non-negotiables

1. **Separate user-content origin.** User-uploaded HTML/SVG/PDF must serve from a registrable origin distinct from the Drive app. Same-origin user content is XSS waiting to happen. ([Google — Content hosting for the modern web, 2012](https://security.googleblog.com/2012/08/content-hosting-for-modern-web.html); [Google — Securely Hosting User Data in Modern Web Applications, 2023](https://security.googleblog.com/2023/04/securely-hosting-user-data-in-modern.html))

2. **No executable user content from the app origin, ever.** No `text/html`, `application/xhtml+xml`, `image/svg+xml`, or `application/pdf` (without explicit attachment-only opt-in) ever leaves `app_origin`. Router refuses; headers refuse; CSP refuses.
